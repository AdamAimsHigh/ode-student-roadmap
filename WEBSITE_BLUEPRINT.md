# WEBSITE_BLUEPRINT.md

> **The definitive master architecture blueprint for the Staples Education ODE Roadmap** — a zero-build, vanilla-JS single-page application. This is the *living blueprint* mandated by the **Architectural Governance & Living Blueprint Rule** in `PIPELINE_LEARNINGS.md`: every structural change to the global web shell (script load order, theme tokens, routing mechanics, or persistence keys) MUST update the corresponding pillar below before a sprint concludes.
>
> It decouples the **core layout systems** (shell, routing, theming, persistence) from the **mathematical data layers** (curriculum, checkpoints, quizzes, practice sets). The two divisions below capture how the shell *got here* (archaeology) and what it *guarantees going forward* (the system spec).

## Division II: Reusable Architecture Pillars (The System Spec)

These five pillars are the load-bearing guarantees of the shell. **Changing any of them obligates an update to this file** (the Living Blueprint Rule).

### Pillar 1: Zero-Dependency Client Architecture
- **Multi-app repository layout (2026-07-01 restructure).** The repository root is the deployed web root of **StaplesEducation.com**: the tutoring business landing page (`index.html`, `reviews.html`, `style.css`) serves at `/`, and the ODE roadmap SPA lives in the standalone directory shell **`ode/`** (renamed from `app/`), served at **`/ode/`**. Deployment target is **Cloudflare Workers static assets** (superseding the original GitHub Pages target): `wrangler.jsonc` points `assets.directory` at the repository root, and the root `.assetsignore` is a **deny-by-default allowlist** re-including only `index.html`, `reviews.html`, `style.css`, `favicon.ico`, `favicon.png`, `fonts/`, and `ode/` — everything else (`node_modules/`, `scripts/`, PDFs at root, the Python toolchain) is shielded from cloud uploads by construction. **Explicit hard-deny + LaTeX-source prune (Bundle 1, 2026-07-05).** Below the credential hard-deny, `.assetsignore` now carries a second belt-and-suspenders block (`scripts/`, `content/`, `src/`, `docs/`, `*.py`, `__pycache__/`, LaTeX build fragments, `unit*_data.json`) so a future broadening of the allowlist can never leak raw curriculum data or the Python/LaTeX toolchain into the upload. Because later rules win, the `*.tex` (and `*.aux`/`*.toc`/`*.out`/`*.fls`/`*.fdb_latexmk`/`*.synctex.gz`) patterns ALSO re-exclude the 59 LaTeX authoring sources that live *under* the re-included `ode/assets/markdowns/` tree — source-only files never fetched at runtime (verified: the sole reference is a `main.css` comment) — trimming edge weight. **Deliberately NOT denied: `*.pdf` and `*.json`** — either glob would win over `!/ode` and silently drop the 58 shipped cheat-sheet PDFs (`ode/assets/pdfs/`) or the SPA data mirrors (`ode/data/*.json`); prune by directory or by a source-only extension, never by those two globs. The `src/worker.js` entry point reaches the edge as the Worker `main` script (bundled by wrangler), never as a served static asset, so it is denied from the asset upload by both the `/*` default and the explicit `src/` rule.
- **Version-control PDF policy (corrected 2026-07-02).** The root `.gitignore` ignores `*.pdf` repo-wide (textbooks and large media stay out of history), with a single re-include exception for the SPA's shipped cheat sheets: `!ode/assets/pdfs/*.pdf` (line 40). The exception originally read `!app/assets/pdfs/*.pdf` and was left dangling by the 2026-07-01 `app/` → `ode/` rename — the 58 already-tracked PDFs were unaffected (tracked files stay tracked regardless of ignore rules), but any *newly added* cheat-sheet PDF would have been silently ignored until the path was corrected in commit `18b841a`. **Rename rule going forward:** any future relocation of the `ode/` tree obligates a same-commit audit of both path-anchored ignore layers — this `.gitignore` exception *and* the `.assetsignore` allowlist above.
- **Root favicon (2026-07-02).** `/favicon.ico` is a committed 1,150-byte 16×16 32-bit ICO placeholder, solid brand purple `#7a00e6` (the landing page's `--color-primary`), added to silence the browser's automatic `/favicon.ico` request that previously 404'd in every visitor's console. Site-wide by construction: no page carries a `<link rel="icon">`, so all apps inherit the root fallback — the SPA's `file://` contract is untouched. **Deployment rule:** the asset ships **only** because of the matching `!/favicon.ico` re-include in `.assetsignore`; any root-level web asset added in the future needs its own `!` entry in the same commit, or the deny-by-default allowlist silently keeps it out of the Cloudflare upload (the deployed 404 persists even though the file is committed). Replacing the placeholder with a real brand mark is a drop-in overwrite plus `npm run deploy`.
- **`file://` executable, no server.** The app must run by opening `ode/index.html` directly — **no Node, no Express, no bundler, no build step.** The `ode/` shell is fully self-contained: every internal reference is document-relative (`css/…`, `js/…`, `assets/pdfs/…`), so the same tree runs identically from `file://`, from `/ode/` in production, or from any future mount point.
- **All first-party code is plain script-injected globals**, never ES modules with `import`/`export` and never a packaged dependency tree. The single source of truth for boot order is the ordered `<script>` block at the bottom of `ode/index.html`:

  ```
  curriculum-data.js   → state.js → theme.js → modes.js
  → checkpoints/checkpoint-registry.js → checkpoint-core.js → widgets-unit0..18.js
  → quiz-data.js → quiz-engine.js → router.js   (router is ALWAYS last)
  ```

  **Load order is a contract:** data and the state machine must be parsed *before* the router that consumes them; the router is loaded last and bootstraps the first render.
- **Data is embedded, never fetched.** Because `file://` blocks `fetch()` of sibling files, every data layer ships as an in-JS global (`CURRICULUM`, `QUIZ_DATA`, `PRACTICE_DATA`, …). This is why the 193-problem practice store was backfilled *into* `quiz-data.js` rather than loaded from `scripts/unitN_data.json` at runtime.
- **External libraries are the only network dependency**, all via CDN `<script>`/`<link>` in `<head>`: **KaTeX 0.16.8** (CSS + JS + `auto-render`), **Math.js 11.8.0** (local parsing), **Desmos v1.9** (interactive checkpoints), and **Google Identity Services** (`accounts.google.com/gsi/client`, async/defer, for the optional cloud sync of Pillar 5 — degrades silently on `file://` or offline). Local stylesheets load `theme.css` **before** `main.css` so custom properties exist before components consume them.
- **Subresource Integrity contract (2026-07-02).** The four immutable versioned CDN artifacts (KaTeX CSS/JS/auto-render 0.16.8 on jsdelivr, Math.js 11.8.0 on cdnjs) carry pinned `integrity="sha384-…"` + `crossorigin="anonymous"` attributes, so a compromised or tampered CDN response is refused by the browser instead of executed — this closes the primary XSS vector against the Pillar 5 localStorage-held credentials. The KaTeX hashes match the officially published 0.16.8 release values. Both CDNs serve `Access-Control-Allow-Origin: *`, so the CORS-mode fetches SRI requires still succeed from the `file://` null origin (Pillar 1 holds). **Deliberately unpinned:** Desmos `v1.9/calculator.js` and `gsi/client` are rolling endpoints redeployed in place by their vendors; a pinned hash would break checkpoints/sign-in on their next release. Version bumps of a pinned library MUST update the matching hash in the same edit.
- **Serverless companion (2026-07-01; hardened + extended 2026-07-03; lead de-dup 2026-07-04).** The only server-side code in the repository is `src/worker.js`, the Worker entry (`wrangler.jsonc` `main`) serving two API routes in front of the asset pipeline. **`/api/sync`** verifies Google ID tokens with the native Web Crypto API against Google's JWKS and reads/writes per-student progress in the `ODE_PROGRESS_KV` KV namespace; its POST path enforces `application/json`, a double size cap (declared `Content-Length` pre-read + actual text post-read, 128 KiB), and top-level-object shape before the whitelist sanitizer runs. **`/api/contact`** (POST only) is the landing page's lead-generation intake: strict payload shape (`name`/`email`/`message`, trimmed + length-capped, email regex), server-side **Cloudflare Turnstile** validation against `challenges.cloudflare.com/turnstile/v0/siteverify` using the `TURNSTILE_SECRET_KEY` **Worker secret** (provisioned via `npx wrangler secret put TURNSTILE_SECRET_KEY` — never in the repo or `vars`), and verified leads persisted as `lead:<iso>_<rand>` KV records with a 90-day TTL. **In-isolate lead de-duplication (2026-07-04):** immediately after Turnstile passes and before the KV write, the handler hashes the normalized `name`/`email`/`message` (SHA-256, email lowercased, NUL-separated) and consults a short-TTL (10-minute) content-hash map held in isolate memory — the same in-isolate, deliberately non-KV pattern as the rate limiter. A hash still present is treated as a rapid duplicate (double-click, or a retry that re-earned a token) and short-circuits with `200 {ok:true, deduped:true}`, cutting no second `lead:` record and firing no second owner email. The hash is recorded **only after** a successful KV write, so a failed write never suppresses the client's legitimate retry; only the blinded hash is held, never lead text; and cross-isolate or post-TTL duplicates fall through to the durable path by design (the goal is collapsing bursts, not global exactly-once). Both routes also sit behind a lightweight in-isolate fixed-window **rate limiter** keyed on `CF-Connecting-IP` (sync 60/min, contact 5/min; 429 + `Retry-After`). **Form-session timing gate (Bundle 1, 2026-07-05):** a stateless, cryptographically verifiable anti-flood layer *under* Turnstile. When the `FORM_SESSION_SECRET` **Worker secret** is provisioned, `handleStorefront` stamps the contact form's hidden `formSessionToken` input (via `HTMLRewriter`) with a token of shape `<issuedAtMs>.<base64url(HMAC-SHA256(issuedAtMs))>`; on submit `verifyFormSessionToken` recomputes the HMAC (constant-time compare) and requires the token's age to clear a realistic human reading window (`>= 2.5 s`) and stay under a `2 h` bound, dropping the instant scripted post or manual flood that a fresh Turnstile token alone would still admit. The check runs *before* the siteverify network round trip (cheapest rejection first) and is **inert unless the secret is set** — no injection, no check — the same graceful-degradation contract as `EMAIL_API_KEY`, so the code can ship dark and be armed later without a flag-day, and a page cached before rollout is never rejected. Stateless by design (no KV), mirroring the rate-limiter and de-dup gates. Regression surface: `scripts/test_sync_worker.mjs` (`npm run test:sync`), **44 cases** — the de-dup collapse, the Geo-SEO sitemap block, the Geo-SEO routing and prototype-pollution guard, and the form-session accept/too-fast/missing/forged boundaries, extended in **Bundle 3** by the parametric subject×modality×location routing (example slugs resolve to the storefront root; unapproved subjects and unknown locations never route), the 200+ sitemap enumeration (every combination listed monthly at 0.9, curated overrides emitted exactly once), and the `lin_alg` subject-track round-trip through its own isolated KV namespace. The Worker is a *companion*, not a dependency: the client never requires it to function (Pillar 1 holds — the SPA remains fully `file://` executable).
- **Scoped KaTeX re-render guard (2026-07-04).** The three client render helpers that drive the browser's KaTeX auto-render pass — `renderPracticeMath` (`ode/js/views-practice.js`), `QuizEngine.renderMath` (`ode/js/quiz-engine.js`), and `CheckpointCore.renderMath` (`ode/js/checkpoints/checkpoint-core.js`) — now pass `ignoredClasses: ["katex"]` to `renderMathInElement`. Because KaTeX's output root carries class `katex`, a repeat pass over a partly-hydrated container walks only the un-rendered text nodes instead of re-descending settled math subtrees on a transient hashchange or widget re-render; freshly injected `$...$` text is never inside a `.katex` node, so nothing new is ever missed. The quiz hint box additionally typesets **once** on first reveal (a `dataset.mathTypeset` one-shot), so hide/show toggling its unchanged content never re-scans it. This is a pure client-side render optimization: no new global, no script-load-order change, no build step — **Pillar 1** (`file://` executable, zero-dependency) is untouched.
- **ODE SEO metadata layer (2026-07-04; a routing-config change, logged per the Living Blueprint Rule).** The roadmap SPA is a single static document at `/ode/`, but its curriculum spans 19 first-principles units, so a crawler hitting `/ode/?unit=N` should read `<title>`/`<meta name="description">` written for that unit's concept rather than the generic shell copy. `src/worker.js` owns a localized `ODE_UNIT_SEO` map (keys `"0"`..`"18"`, matching `content/curriculum.json` unit numbers) and a `handleOdeSeo()` handler that fetches the real asset and **stream-rewrites only those two head tags via `HTMLRewriter`** when a recognized `?unit=` selector is present; every miss (no selector, unknown unit, non-GET, non-HTML) returns the untouched asset, so this is a purely additive layer. **The load-bearing routing change:** the handler can only run because `wrangler.jsonc` now lists `"run_worker_first": ["/ode/", "/ode/index.html"]` in `assets`, routing *only* the ODE document through the Worker before the asset server; sub-assets (`/ode/css`, `/ode/js`, `/ode/data`) stay on the direct asset path with zero Worker overhead. SEO copy obeys the §1 constraint (no em-dashes, no ampersands). **Prototype-blank hardening (2026-07-06):** `ODE_UNIT_SEO` is now a frozen prototype-blank dictionary (`Object.freeze(Object.assign(Object.create(null), {…}))`), so the previously unguarded `ODE_UNIT_SEO[unitParam]` lookup — driven by the attacker-controlled `?unit=` selector — can no longer resolve `constructor`/`__proto__`/`toString`/`hasOwnProperty` to an inherited `Object` member and mint a phantom unit route with `undefined` metadata; an unknown selector returns `undefined` and the untouched asset, exactly as before for real misses. The SPA's zero-dependency `file://` contract is untouched — the rewrite happens edge-side and `file://` opens never traverse the Worker. **Crawlability (2026-07-05):** the dynamic `/sitemap.xml` handler (`buildSitemapXml()`) now enumerates the two top-level pages **plus one deep-linked `<url>` per unit** (`/ode/?unit=N`, weekly `changefreq`), drawing the unit list from `ODE_UNIT_SEO` itself so a new unit self-registers in the sitemap; every `<loc>` is XML-entity-escaped. So the 19 unit-targeted metadata variants are not just correct-on-hit but independently discoverable by search engines. **Per-unit `<lastmod>` (2026-07-05):** each `ODE_UNIT_SEO` block now carries an ISO-8601 `lastmod` (the git last-commit date of that unit's content package source `scripts/unitN_data.json`), which `sitemapUrlEntry()` emits per unit and falls back to the global `SITEMAP_LASTMOD` baseline only if a block omits it — so the sitemap reports genuine per-unit content history (units 1 and 14-18 predate the Content API v1 batch) rather than one blanket date.

The form embeds the implicit-render Turnstile widget (`.cf-turnstile`, public site key `0x4AAAAAADvT-1PlPvun0Xm_`) and surfaces success/error states through the new `--color-success`/`--color-error` (+ `-soft`) tokens added to the `style.css` `:root` block. **Cloudflare Web Analytics** is injected site-wide on the landing page via the deferred `static.cloudflareinsights.com/beacon.min.js` script (public token `14c519923cd24565bcbd4b336c6bb429`, cookie-free). The `ode/` SPA is untouched by all of this — its zero-dependency `file://` contract is unchanged.

Both scroll smoothly with **zero JavaScript** — they rely on the existing `html { scroll-behavior: smooth }` in `style.css`, so they behave identically over `file://` and `https`. All `.booking*` styles are new to `style.css` and reuse the established surface/border/purple-halo tokens (same language as `.card`/`.card-icon`); the copy honors the §1 constraint.
- **Progressive Web App layer (Bundle 2, 2026-07-05; a web-shell addition, logged per the Living Blueprint Rule).** The `ode/` SPA now ships an installable PWA surface as a **pure progressive enhancement** that leaves Pillar 1 intact. Two new self-contained files live beside the shell: **`ode/manifest.json`** (brand metadata: name "Staples Education ODE Roadmap", short_name "ODE Roadmap", `theme_color #7a00e6`, `background_color #000000`, `display standalone`, `start_url`/`scope` set to `.` so it is mount-point agnostic, and the icon array pointing to the root transparent master `/favicon.png` at 192 and 512) and **`ode/sw.js`**, a dependency-free service worker implementing **cache-first with network fallback**. On install it precaches a deliberately small, guaranteed-present core shell (`./`, `./index.html`, the two stylesheets, and the boot JS `curriculum-data.js`/`state.js`/`router.js`) tolerating individual misses so one unavailable asset never aborts the install; a runtime cache then lazily stores any other same-origin asset plus the two **pinned, immutable math CDNs** (`cdn.jsdelivr.net` KaTeX, `cdnjs.cloudflare.com` Math.js) on first fetch, so a revisited workbook opens with zero network latency and survives offline (navigations fall back to the cached `./index.html`). API traffic (`/api/*`) and the rolling third-party scripts (Desmos, Google Identity, analytics) are **never cached** — always live. **`file://` contract preserved two ways:** the worker carries a `DISK_RUN` guard (`self.location.protocol === "file:"`) that makes every lifecycle handler inert, and the registration snippet added to `ode/index.html` is gated on **both** service-worker support **and** an `https:` origin (`window.location.protocol !== "https:"` bails), so a disk open — or any insecure/localhost-http origin — registers nothing and the app stays a plain zero-dependency SPA. The service worker scope is `./` (i.e. `/ode/`), so it can never intercept the root storefront or its edge-rewritten HTML.
- **Local Geo-SEO landing matrix (Bundle 1, 2026-07-05; a routing-config change, logged per the Living Blueprint Rule).** `src/worker.js` owns a frozen `GEO_SEO_REGISTRY` mapping clean, keyword-anchored neighborhood slugs (`scottsdale-calculus-tutor`, `paradise-valley-math-tutor`, `gilbert-math-tutor`, `chandler-math-tutor`, `tempe-calculus-tutor`, `arcadia-math-tutor`, `ahwatukee-math-tutor`, `fountain-hills-math-tutor`) to a localized `{ title, description, headline, subhead }`. No new page files, no template duplication, and the on-disk landing template is untouched — the rewrite happens only at the edge, so Pillar 1's `file://` contract holds. Registry copy obeys the §1 constraint (no em-dashes, no ampersands). Lookups go through `geoRegistryEntry()` (an `hasOwnProperty` guard) so `/constructor`, `/toString`, etc. can never resolve to an inherited `Object` member and mint a phantom route. **(2026-07-06):** the registry is additionally instantiated on a prototype-blank base (`Object.freeze(Object.assign(Object.create(null), {…}))`), so the `hasOwnProperty` guard and the null prototype are now belt-and-suspenders — the whole geo lookup surface (curated `GEO_SEO_REGISTRY`, the parametric `SEO_LOCATION_BY_SLUG` `Map`, and the frozen `SEO_SUBJECTS`/`SEO_NEIGHBORHOODS` arrays) is uniformly free of inherited-property routes. **The load-bearing routing change:** `wrangler.jsonc` `assets.run_worker_first` now lists `"/"` and `"/index.html"` in addition to the ODE entries, so the storefront document traverses the Worker before the asset server — this is what lets `handleStorefront` also stamp the contact form's hidden `formSessionToken` (see the serverless-companion bullet) on the canonical landing page, not just the geo variants. Geo slugs are not assets, so they reach the Worker via the normal fallback and are intercepted just ahead of the static-asset line. Sub-assets (`/style.css`, `/fonts`, `/ode/*`) stay on the direct asset path with zero Worker overhead. Because the injected timing token is per-render, `handleStorefront` sets `Cache-Control: no-store` on the rewritten HTML document (only the doc; sub-assets are unaffected). **Crawlability:** `buildSitemapXml()` appends one `<url>` per registry slug drawn from `GEO_SEO_REGISTRY` itself (so a new neighborhood self-registers), each pinned to `changefreq monthly` and an elevated `priority 0.9`, `<loc>` XML-escaped like every other entry. The corresponding storefront-template change is the single hidden `<input name="formSessionToken">` added to the `#contact` form in `index.html` — zero new dependency, protocol-agnostic (empty and harmless under `file://` or when the secret is unset).
- **Parametric Geo-SEO matrix (Bundle 3, 2026-07-05; a routing-config change, logged per the Living Blueprint Rule).** Beneath the curated `GEO_SEO_REGISTRY` override layer, `src/worker.js` now owns a **parametric keyword-registry matrix** that mints a landing page for every combination of an approved **subject** and a **modality**. Two frozen immutable arrays are the source of truth: **`SEO_SUBJECTS`** (13 approved subjects — ACT Math, Algebra 1, Algebra 2, Calculus, Differential Equations, Discrete Math, Finite Math, Linear Algebra, Prealgebra, Precalculus, SAT Math, Statistics, Trigonometry) and **`SEO_NEIGHBORHOODS`** (15 Phoenix-metro neighborhoods). The two **modalities** (`SEO_MODALITIES`) are *Local Neighborhoods* (fans out over every neighborhood token, slug `<neighborhood>-<subject>-tutor`, e.g. `/scottsdale-linear-algebra-tutor`) and *Online and Remote* (the single `online` token, slug `online-<subject>-tutor`, e.g. `/online-discrete-math-tutor`). That is 13 × (15 + 1) = **208 parametric combinations**; de-duplicated against the 8 curated overrides that share the slug space (`scottsdale-calculus-tutor`, `tempe-calculus-tutor`), the matrix yields **214 discrete landing URLs** (235 total `<url>` blocks with the 2 top pages + 19 units). **Routing:** a unified **`resolveGeoMeta(slug)`** checks the curated registry first, then `parametricGeoMeta()`, and drives both the router intercept (just ahead of the static-asset fallback) and `handleStorefront`'s `HTMLRewriter` injection — so a parametric hit is served the **same root `index.html` template**, stream-rewritten edge-side with a per-pairing `<title>`, `<meta name="description">`, and hero `.hero-content h1`/`p` (0 ms server lag, no new page files, on-disk template untouched → Pillar 1 holds). **Parse safety:** because both subjects and locations contain hyphens, `parametricGeoMeta` anchors on the `-tutor` suffix and matches an approved subject as a suffix of the remainder, requiring the leading segment to be a **registered location token looked up through a `Map`** (never a bare object), so `/constructor-…`, unapproved subjects (`/scottsdale-biology-tutor`), and unknown locations (`/atlantis-calculus-tutor`) all fall through to the asset server and never mint a phantom route. Generated copy is built by `buildParametricMeta` and obeys the §1 constraint (no em-dashes, no ampersands; commas and "and"). **Crawlability:** `buildSitemapXml()` now iterates **`allGeoSlugs()`** — curated slugs first, then every parametric combination drawn from the frozen registries themselves (so a new subject or neighborhood self-registers), each pinned to `changefreq monthly` + `priority 0.9`, a curated override emitted exactly once.

### Pillar 2: Centralized Hash Routing Matrix
- **One entry point.** `ode/js/router.js` exposes `renderCurriculum()`, the sole render function. Every navigation, mode toggle, and checkpoint pass funnels through it, so the view always redraws to match the URL hash.
- **Subpath mounting (2026-07-01 restructure).** In production the SPA mounts at `/ode/`; the hash matrix below is unchanged because every route lives *after* the `#` and every asset link is document-relative, so the router is agnostic to its mount point. Cross-app navigation is one-way by design: the landing page and `reviews.html` link **into** the SPA via the `/ode/` primary CTA in their shared navigation header; the SPA contains no root-absolute links back out, preserving its standalone `file://` contract.

  - `document.addEventListener("DOMContentLoaded", renderCurriculum)` — initial paint.
  - `window.addEventListener("hashchange", …)` — re-render plus `window.scrollTo(0, 0)` on view changes.

### Pillar 3: The Unified Data Model Separation
- **The canonical content layer (2026-07-03, Content API v1).** All curriculum, quiz, and practice content is canonically owned by the **`content/` tree** — `manifest.json` + `curriculum.json` + per-unit packages `content/units/unit-NN/{guide,practice,quizzes}.json` — specified by `docs/content-api/CONTENT_API_SPEC.md` and its JSON Schemas. The script-injected globals below are **generated artifacts**: `scripts/compile_web.py` compiles `content/` into `ode/js/curriculum-data.js`, `ode/js/quiz-data.js`, and both `ode/data/*.json` mirrors (drift is now impossible by construction; every generated file carries a banner). CI shape: `python scripts/validate_content.py && python scripts/test_content_pipeline.py` (the latter includes `compile_web.py --check` staleness detection and byte-exact print-reconstruction locks). **Edit `content/`, never the generated files.** The print pipeline reads the same canon: `render_latex.py` loads units via `scripts/content_loader.py` (semantic macros `\emph/\strong/\highlight/\work/\warn` lower per target); `build_practice_data.py` is retired. The legacy `scripts/unitN_data.json` files are frozen archival migration sources.
- **Layout never embeds content.** The curriculum map, checkpoint widgets, quiz strings, and practice problems are each isolated in their own script-injected global, consumed by the layout files but never authored inside them:
  - `curriculum-data.js` → `CURRICULUM` (Unit → Module → Video → `interactive_checkpoint`), generated from `content/curriculum.json` (canonical names `videoId`/`checkpoint` are lowered to the legacy `video_id`/`interactive_checkpoint` until the router migrates).
  - `checkpoints/checkpoint-registry.js` + `widgets-unitN.js` → per-unit interactive widget definitions, resolved by the `interactive_checkpoint` key on each module.
  - `quiz-data.js` → `QUIZ_DATA` (micro-practice + mastery) **and** `PRACTICE_DATA` (the web practice store, units 0–18). `unit_mastery` stays keyed by exact unit title at emission (legacy router contract); canonically it is keyed by `unitNumber`.
- **The renderer is data-driven.** `router.js` reads these globals and builds DOM with shared classes (`.practice-problem-list`, `.practice-solution-list`, `.pdf-download-btn`, `.toc-grid`, …). Adding a unit's content requires no router change — only a data entry.
- **Build-time tooling produces data, never runtime behavior.** `scripts/compile_web.py` (web globals + mirrors, superseding the retired `build_practice_data.py`) and `scripts/render_latex.py` (PDF assets, via `content_loader.py`) are dev-time compilers over the canonical `content/` tree. The web compiler lowers canonical strings into web-clean text (`$…$` / `$$…$$`, semantic macros unwrapped, print spacing stripped) and emits static globals, preserving Pillar 1.
- **The built-sandbox model is rich, not a placeholder.** Where most units fall back to the generic `INTERACTIVE_SANDBOXES` placeholders (one static "coming soon" body per unit), the leading units are served by dedicated programmatic arrays in `router.js`: `UNIT_0_SANDBOXES` (nine entries), `UNIT_1_SANDBOXES` (ten entries), and `UNIT_2_SANDBOXES` (10 entries, the complete first-order toolkit: linear-solver, geometric-transformation, applied-physics, and the master differential-matrix workbench). Each entry carries a stable, **descriptive ID slug** (`unit_0_*`, `unit_1_*`, `unit_2_*` — e.g. `unit_2_integrating_factor_builder`), dashboard copy, and an explicit `render(body)` function rather than an empty body, so the dashboard loop mounts a live, self-contained canvas engine. Every engine runs a **self-terminating `requestAnimationFrame` loop** that stops once its canvas leaves the DOM, uses strict per-card id/variable namespacing (`u0_sN_`, `u1_sN_`, and for Unit 2's cluster `u2_s1_` through `u2_s10_`) so no inner id or state can collide globally, and reads its colors live from the Pillar 4 theme tokens so Light and Dark both render natively. Registration is uniform: each array is folded into `buildInteractiveItems()` under the same `900 + index` sandbox sort band, and each card deep-links by its slug at `#interactives-sandbox-<id>`. The data-driven `isSandbox` path is unchanged — the renderer simply invokes the entry's `render` when present — keeping layout free of engine logic.
- **`SandboxKit` — the templatized canvas-playground wrapper (Bundle 3, 2026-07-05).** The ~30 engines above each hand-roll the identical scaffold: create a canvas + 2D context, read theme tokens live at draw time, and run the self-terminating `requestAnimationFrame` loop guarded by `document.body.contains(canvas)`. **`SandboxKit`** (top of `ode/js/sandboxes.js`) factors exactly that boilerplate into one data-driven mount — `SandboxKit.mount(body, { width, height, draw(ctx, env) })`, where `env = { color, canvas, w, h }` — so a **new** sandbox is a small config object plus a pure `draw` function operating in CSS pixels (the kit owns DPR scaling, live theme reads via `env.color(token, fallback)`, and frame teardown; a throwing `draw` is caught once and halts the loop rather than spamming every frame). This turns "deploy the next unit's interactive" from another 150-line copy of the scaffold into a **data-entry mapping**. The existing `u0_/u1_/u2_` engines predate the kit and are **deliberately left untouched** (incremental adoption — no churn on working code); upcoming units adopt it. Zero dependency, `file://` safe (pure DOM + canvas), no module system or script-load-order change — **Pillar 1** holds.
- **Authoring-pipeline consolidation (Bundle 3, 2026-07-05).** The canonical **Content API v1** (`content/` tree → `scripts/compile_web.py` → web globals + mirrors) is the single build-time content path; the legacy generators fold into it rather than writing shipped globals directly. `generate_micro_quizzes.py` (`--video-id`, `--model` flags; Gemini-driven quiz drafting from YouTube transcripts, the original Phase 1 pipeline) and the per-unit `scripts/gen_units*_quizzes.py` batch drafters are **content authoring aids whose output is hand-reconciled into `content/units/unit-NN/quizzes.json`** (the `micro_practice` + `unit_mastery` blocks defined by `docs/content-api/CONTENT_API_SPEC.md`), then compiled into `ode/js/quiz-data.js` by `compile_web.py` — never edited into the generated file. So a generator's role ends at *drafting a `content/` JSON fragment*; canon, validation (`validate_content.py`), and emission stay in the one pipeline, keeping drift impossible by construction (Pillar 3's "edit `content/`, never the generated files").

### Pillar 5: Quiz Subsystem Persistence Loop
- **All progress serializes to discrete, namespaced `localStorage` keys** through the `ODEState` IIFE in `ode/js/state.js`. Every key is `ode_*`-prefixed:

  | Key | Holds |
  |---|---|
  | `ode_watched_videos` | array of watched `video_id`s |
  | `ode_passed_checkpoints` | map of `checkpointId → result detail` (see checkpoint-detail schema below) |
  | `ode_quiz_progress` | map of `quizId → [correctly-answered question ids]` |
  | `ode_learning_mode` | `"exploration"` (default) or `"guided"` |
  | `ode_theme_preference` | `"light"` (default) / `"dark"` / `"system"` |

- **Checkpoint-detail schema (Bundle 2, 2026-07-05; scoped slider persistence).** Each `ode_passed_checkpoints[checkpointId]` value is a free-form detail object persisted verbatim; `setCheckpointPassed` writes `{ passed: true, attempts, checkpoint }` plus, when the checkpoint owns sliders, up to two coordinate maps: **`sliders`** (`{ sliderLabel: finalNumericValue }`) for native `rangeControl` range inputs, and **`desmosSliders`** (`{ latexVariable: finalNumericValue }`) for Desmos calculator variables watched via `observeValue`. Both are captured **exactly once, at pass time** (`CheckpointCore.snapshotSliders`) rather than on every drag, so the size-capped cloud snapshot is never flooded by real-time input churn. `CheckpointCore` owns a build-time `activeSliderCapture` context that `shell()` opens around `buildFn` (and always restores in a `finally`), holding two registries — one per family. **Hydration on rebuild of a passed checkpoint** (both read the saved detail via the `ODEState.getCheckpointDetail(id)` accessor): a `rangeControl` restores its saved value when it is within the slider's `[min, max]`, else the configured default; an `observeValue` variable is **re-injected straight into the live Desmos instance** (`setDesmosVariable` → `calculator.setExpression`) by locating the existing expression that defines that variable and updating it **by id**, so the slider definition and its `sliderBounds` survive (a bare `C=<val>` would collide with the widget's own `C=1` and error), and only when the saved value sits inside those bounds. A checkpoint with only native sliders, only Desmos sliders, neither, or a Desmos widget that failed to load (`observeValue(null)`) degrades cleanly: absent families simply write no key and throw nothing. Server-side this needs **no Worker change** — the whole detail object is already whitelisted and JSON-size-capped (2 KiB) by `sanitizeProgress`, and the union-merge boot logic (checkpoint entries added only when locally absent, local detail wins) preserves both saved coordinate maps across devices unchanged.
- **Defensive I/O.** `readJSON`/`writeJSON` wrap every access in `try/catch` with a fallback, so corrupt or unavailable storage degrades gracefully instead of throwing.
- **Domain semantics.** A quiz's score *is* the length of its correct-answer array; `clearQuizProgress(quizId)` powers per-quiz "Retry" without touching siblings. `resetAllProgress()` clears watched videos, checkpoints, and quiz progress **but deliberately preserves** `ode_learning_mode` and `ode_theme_preference` (UI preferences survive a progress reset).
- **Mode logic (ARCHITECTURE.md §4).** *Exploration* (default) leaves everything unlocked; checkpoints are optional self-assessments. *Guided Pathway* activates gateways: the next module unlocks only after the current checkpoint passes, enforced via the Pillar 2 flat-offset sequence. On failure the system surfaces a first-principles guiding question, never the answer.
- **Cloud sync loop (2026-07-01, optional layer over the localStorage loop; 2026-07-02 long-lived hybrid local/edge token schema).** When a Google Identity Services credential or a Worker-issued edge session is active, `state.js` extends the persistence loop to the `/api/sync` endpoint (root-absolute by necessity — the SPA mounts at `/ode/` while the API lives at the domain root; a protocol guard keeps it inert under `file://`):
  - **Credentials are a two-tier hybrid, both persisted in namespaced `localStorage` keys** (superseding the original session-scoped `sessionStorage` cache, so sign-in survives browser restarts on mobile and desktop):
    | Key | Holds | Lifetime authority |
    |---|---|---|
    | `ode_google_credential` | the raw GIS ID token (RS256 JWT) | Google (`exp` claim, ~1 h; expired tokens dropped on read) |
    | `ode_cloud_session` | `{ token, expiresAt, email }` — the opaque `odesess_*` edge session token | the Worker's KV record (30-day TTL); the local `expiresAt` is only a skip-a-doomed-request hint |
  - **Token exchange:** the first request authenticated with a verified Google ID token is answered with a freshly minted `odesess_*` session token (`session` field on the sync response), which the client adopts and prefers for all subsequent requests. A 401 on a session token clears it and retries once with the Google credential; with neither available the auth UI resets (sign-in surfaces restored via `syncGoogleSurfaces()`) and GIS `auto_select: true` silently re-establishes the account on the next load.
  - **Defensive boot:** `bootCloudSync()` reads the cached session (or a still-valid credential) from `localStorage` on script parse and initializes the cloud loop immediately; corrupt or expired entries fall through to the signed-out state. The GIS callback global is `odeHandleGoogleCredential`; sign-out clears both keys and calls `google.accounts.id.disableAutoSelect()`.
  - **Auth surface lifecycle (2026-07-02).** GIS is initialized **programmatically** (`initGoogleIdentity()` in `state.js`, wired to the SDK's documented `onGoogleLibraryLoad` hook, with a boot-time fallback call). The declarative `g_id_onload`/`g_id_signin` markup was removed: it rendered the embedded button and fired the floating One Tap prompt unconditionally as soon as the async SDK landed, re-showing sign-in chrome to already-authenticated students. `syncGoogleSurfaces()` is the single gate — only a **completely anonymous** visitor (no `ode_cloud_session`, no still-valid `ode_google_credential`) gets the button rendered into `#auth-signin-slot` and a `google.accounts.id.prompt()` One Tap. A signed-in boot hides the slot via the `hidden` attribute, enforced by an `!important` CSS guard (`#auth-signin-slot[hidden]`); a fresh sign-in additionally calls `google.accounts.id.cancel()` to dismiss any floating prompt. Sign-out restores both surfaces (slot unhidden, button rendered on demand, One Tap re-prompted) with `disableAutoSelect()` guaranteeing an account chooser rather than a silent re-login. Every GIS call is wrapped so `file://` and offline sessions no-op silently.
  - **Boot merge:** on activation the cloud record is fetched and **defensively merged** — watched videos and per-quiz correct-answer lists are unioned, checkpoint entries are added only when locally absent (local detail wins) — so a stale device can never erase work done elsewhere. A changed merge triggers one `renderCurriculum()` repaint.
  - **Background push:** every progress mutator (`setVideoWatched`, `setCheckpointPassed`, `setQuizAnswerCorrect`, `clearQuizProgress`, `setLearningMode`, `resetAllProgress`) schedules a silent, debounced (2.5 s) POST of the whitelisted `ode_*` snapshot; `ODEState.requestCloudSync()` exposes the same trigger for widgets that persist through their own pathways. All cloud calls are best-effort `try/catch` — the localStorage loop is authoritative and never blocks on the network.
  - **Server contract:** `src/worker.js` (Pillar 1 serverless companion) intercepts `GET`/`POST` `/api/sync` and resolves the bearer in one of two verification modes before touching any `progress:<sub>` record: an `odesess_*` token validates directly against its KV session record (`session:<sha256(token)>` — only the hash is stored, so a KV dump can never be replayed; TTL expiry yields a 401 challenge), while anything else must pass the full edge Google JWT verification (JWKS signature, `iss`/`aud`/`exp`/`nbf`/`sub`) and is then exchanged for a new 30-day session. Progress writes sanitize every payload against a strict whitelist with size caps. The contract is regression-locked by the committed cryptographic validation suite `scripts/test_sync_worker.mjs` (real RS256 tokens against a mocked JWKS + TTL-aware KV mock; run `npm run test:sync`).
  - **Multi-subject track isolation (Bundle 3, 2026-07-05).** Progress is namespaced by curriculum: the KV key shape is `progress:<subject>:<google sub>`, `subject ∈ KNOWN_SUBJECTS = ["ode", "lin_alg", "calc"]`, selected by the `?subject=` query param (absent → `ode`). **The identity layer is deliberately subject-agnostic** — one `session:<hash>` record and one Google sign-in span every track (SSO preserved), and only progress data is track-scoped, so adding a subject is a one-line registry append and can never fork a student's identity. The **`lin_alg` token is primed end to end**: a write to `?subject=lin_alg` lands in `progress:lin_alg:<sub>`, an `ode` read never sees it, and the ode-only lazy legacy migration (`progress:<sub>` → `progress:ode:<sub>`) does not fire for it — locked by the "parallel subject tracks are isolated namespaces" and "lin_alg track round-trips" cases. Client side, a future Linear Algebra SPA ships the same `state.js` coordinator with its own `SYNC_SUBJECT = "lin_alg"` constant and `lin_alg_*`-prefixed `localStorage` keys (the ODE app stays `SYNC_SUBJECT = "ode"` with `ode_*` keys — complete `localStorage` isolation by prefix). **Known follow-up (not yet built):** `sanitizeProgress` currently whitelists the `ode_*` progress keys specifically; standing up a real `lin_alg` track means generalizing the sanitizer to that subject's key shape. The routing, namespacing, and KV isolation are done and tested; the per-subject *payload schema* generalization is the remaining step when the `lin_alg` workbook actually ships.

## Headless Browser Test Harness (Bundle 4, 2026-07-05)

A Playwright end-to-end harness regression-locks the **student-facing UX state flows** that the unit-test tier (`scripts/test_sync_worker.mjs`, which locks the Worker's crypto/KV contract) cannot see. It drives a real headless Chromium against the production assets and asserts on rendered DOM, animation, and persistence — never mutating a shipped file, so **Pillar 1 holds** (the app is exercised exactly as deployed).

- **Dev-only, zero production footprint.** `@playwright/test` is a dev harness; it never ships to the client and adds no runtime dependency. The `.assetsignore` deny-by-default allowlist (`/*` then re-includes) excludes `tests/`, `playwright.config.js`, `test-results/`, and `playwright-report/` from any Cloudflare upload **by construction** — nothing here can reach the edge. Test output dirs are git-ignored.
- **Local static emulation (`playwright.config.js`).** A `webServer` block spins up Python's stdlib `http.server` over the repository root (the same lightweight server used for every prior local verification pass — no new install), giving specs an `http://127.0.0.1:4173` origin where `/` is the landing storefront and `/ode/` is the SPA, mirroring the Worker asset pipeline's routing. Running over `http` (not `file://`) exercises the http-context code paths; the `file://` zero-dependency invariant remains a **separate** guarantee owned by the app and its other verification passes.
- **Chromium-class boundary.** One `chromium` project is provisioned locally; the mobile-viewport check rides the same engine via a Pixel 5 device profile (`defaultBrowserType` stripped, since it is illegal inside a `describe`). WebKit/Firefox projects can be appended to `playwright.config.js` once their binaries are installed (`npx playwright install`).
- **Lifecycle spec (`tests/roadmap-lifecycle.spec.js`), four critical-path flows:**
  1. **Mobile responsive** — the landing `header.navbar` navigation stacks on a phone viewport with **no horizontal overflow** (`scrollWidth − clientWidth ≤ 1px`) and no clipping (every nav link's box sits inside the viewport width).
  2. **Dashboard hydration** — `/ode/` paints all **19** `.toc-card` unit blocks (each with a non-empty `.toc-card-title`), the DOM contract behind the Pillar 2 Table of Contents.
