# WEBSITE_BLUEPRINT.md

> **The definitive master architecture blueprint for the Staples Education ODE Roadmap** — a zero-build, vanilla-JS single-page application. This is the *living blueprint* mandated by the **Architectural Governance & Living Blueprint Rule** in `PIPELINE_LEARNINGS.md`: every structural change to the global web shell (script load order, theme tokens, routing mechanics, or persistence keys) MUST update the corresponding pillar below before a sprint concludes.
>
> It decouples the **core layout systems** (shell, routing, theming, persistence) from the **mathematical data layers** (curriculum, checkpoints, quizzes, practice sets). The two divisions below capture how the shell *got here* (archaeology) and what it *guarantees going forward* (the system spec).

## Division II: Reusable Architecture Pillars (The System Spec)

These five pillars are the load-bearing guarantees of the shell. **Changing any of them obligates an update to this file** (the Living Blueprint Rule).

### Pillar 1: Zero-Dependency Client Architecture
- **Multi-app repository layout (2026-07-01 restructure).** The repository root is the deployed web root of **StaplesEducation.com**: the tutoring business landing page (`index.html`, `reviews.html`, `style.css`) serves at `/`, and the ODE roadmap SPA lives in the standalone directory shell **`ode/`** (renamed from `app/`), served at **`/ode/`**. Deployment target is **Cloudflare Workers static assets** (superseding the original GitHub Pages target): `wrangler.jsonc` points `assets.directory` at the repository root, and the root `.assetsignore` is a **deny-by-default allowlist** re-including only `index.html`, `reviews.html`, `style.css`, `favicon.ico`, and `ode/` — everything else (`node_modules/`, `scripts/`, PDFs, the Python toolchain) is shielded from cloud uploads by construction.
- **Version-control PDF policy (corrected 2026-07-02).** The root `.gitignore` ignores `*.pdf` repo-wide (textbooks and large media stay out of history), with a single re-include exception for the SPA's shipped cheat sheets: `!ode/assets/pdfs/*.pdf` (line 40). The exception originally read `!app/assets/pdfs/*.pdf` and was left dangling by the 2026-07-01 `app/` → `ode/` rename — the 58 already-tracked PDFs were unaffected (tracked files stay tracked regardless of ignore rules), but any *newly added* cheat-sheet PDF would have been silently ignored until the path was corrected in commit `18b841a`. **Rename rule going forward:** any future relocation of the `ode/` tree obligates a same-commit audit of both path-anchored ignore layers — this `.gitignore` exception *and* the `.assetsignore` allowlist above.
- **Root favicon (2026-07-02).** `/favicon.ico` is a committed 1,150-byte 16×16 32-bit ICO placeholder, solid brand purple `#7a00e6` (the landing page's `--color-primary`), added to silence the browser's automatic `/favicon.ico` request that previously 404'd in every visitor's console. Site-wide by construction: no page carries a `<link rel="icon">`, so all apps inherit the root fallback — the SPA's `file://` contract is untouched. **Deployment rule:** the asset ships **only** because of the matching `!/favicon.ico` re-include in `.assetsignore`; any root-level web asset added in the future needs its own `!` entry in the same commit, or the deny-by-default allowlist silently keeps it out of the Cloudflare upload (the deployed 404 persists even though the file is committed). Replacing the placeholder with a real brand mark is a drop-in overwrite plus `npm run deploy`.
- **`file://` executable, no server.** The app must run by opening `ode/index.html` directly — **no Node, no Express, no bundler, no build step.** The `ode/` shell is fully self-contained: every internal reference is document-relative (`css/…`, `js/…`, `assets/pdfs/…`), so the same tree runs identically from `file://`, from `/ode/` in production, or from any future mount point.
- **All first-party code is plain script-injected globals**, never ES modules with `import`/`export` and never a packaged dependency tree. The single source of truth for boot order is the ordered `<script>` block at the bottom of `ode/index.html`:

  ```
  curriculum-data.js → subject-config.js → telemetry.js → state.js → theme.js → modes.js
  → checkpoints/checkpoint-registry.js → checkpoint-core.js → widgets-unit0..18.js
  → readings-data.js → bank-loader.js → quiz-engine.js
  → charts.js → adaptive.js                       (Day 3 capstone engines, 2026-07-11)
  → sandboxes.js → views-materials.js → views-interactives.js → views-practice.js
  → views-quizzes.js → views-dashboard.js → views-settings.js
  → router.js   (router is ALWAYS last)
  ```

  (2026-07-11, Day 3 capstone: `charts.js` is the dependency-free canvas chart kernel and `adaptive.js` the session composer; both parse after the data shells and before the view layer. `views-dashboard.js` and `views-settings.js` join the view files ahead of the router. All cross-module couplings — the quiz summary's branching panel, the unit detail's remediation detour, the settings role probe — are runtime `typeof`-guarded, so each module degrades silently without its partners.)

  (2026-07-10, Content Schema v2: the monolithic `quiz-data.js` — 2.5 MB parsed at boot — is retired from this block. `readings-data.js` is the tiny eager supplemental-readings global; `bank-loader.js` declares the `QUIZ_DATA`/`PRACTICE_DATA` global shells the views consume and lazily hydrates them per unit, see the lazy-layer bullet below.)

  **Load order is a contract:** data and the state machine must be parsed *before* the router that consumes them; the router is loaded last and bootstraps the first render. `subject-config.js` sits immediately after the generated `curriculum-data.js` because it bridges the `CURRICULUM` global into `SUBJECT_CONFIG.units` (see the Universal SPA Content Matrix Specification below), and `state.js` reads `SUBJECT_CONFIG.subjectId` for its sync namespace, so the config must be parsed before both. `telemetry.js` (2026-07-10) sits immediately before `state.js` because `state.js` reads `ODETelemetry.KEYS` at parse time to fold the telemetry layers into its snapshot, merge, and reset loop; the runtime coupling is guarded on both sides so either module degrades gracefully without the other.
- **Data is embedded, never fetched.** Because `file://` blocks `fetch()` of sibling files, every data layer ships as an in-JS global (`CURRICULUM`, `QUIZ_DATA`, `PRACTICE_DATA`, `READINGS_DATA`, …) — no consumer ever issues a runtime `fetch()` for content.
- **Lazy bank layer (2026-07-10, Content Schema v2 / Sprint Rec 2).** The one sanctioned *deferred* data path: quiz, practice, and adaptive-pool content ships as one generated chunk per unit (`ode/js/bank/bank-unit-NN.js`, dictionary-compressed, ~50–110 KB) instead of a 2.5 MB boot-time monolith. `ode/js/bank-loader.js` owns the `QUIZ_DATA`/`PRACTICE_DATA` shells and `ODEBank.ensureUnit(n, cb)`: the first navigation touching unit *n* injects that chunk via a **dynamic `<script>` element — which executes normally from `file://` where `fetch()` is blocked** — and the view re-renders through the router when it lands (loading/failure notes hold the slots; a failed chunk is sticky for the session so a missing file can never render-loop). Each chunk calls `ODEBank.registerUnit(n, {v, d, p})`: one regex pass restores the compile-time dictionary codes (`U+00A4` sentinel + 2-char base-62 index; the sentinel is validator-forbidden in content), and parametric bank templates are instantiated once per session by a built-in ~40-line arithmetic evaluator (no Math.js dependency, so templates work with every CDN blocked). Chunk `src` values are document-relative, preserving mount-point agnosticism. The Pillar 1 contract is unchanged: classic scripts only, no ES modules, no fetch, fully `file://` executable — regression-locked by smoke-suite core E.
- **External libraries are the only network dependency**, all via CDN `<script>`/`<link>` in `<head>`: **KaTeX 0.16.8** (CSS + JS + `auto-render`), **Math.js 11.8.0** (local parsing), **Desmos v1.9** (interactive checkpoints), and **Google Identity Services** (`accounts.google.com/gsi/client`, async/defer, for the optional cloud sync of Pillar 5 — degrades silently on `file://` or offline). Local stylesheets load `theme.css` **before** `main.css` so custom properties exist before components consume them.
- **Subresource Integrity contract (2026-07-02).** The four immutable versioned CDN artifacts (KaTeX CSS/JS/auto-render 0.16.8 on jsdelivr, Math.js 11.8.0 on cdnjs) carry pinned `integrity="sha384-…"` + `crossorigin="anonymous"` attributes, so a compromised or tampered CDN response is refused by the browser instead of executed — this closes the primary XSS vector against the Pillar 5 localStorage-held credentials. The KaTeX hashes match the officially published 0.16.8 release values. Both CDNs serve `Access-Control-Allow-Origin: *`, so the CORS-mode fetches SRI requires still succeed from the `file://` null origin (Pillar 1 holds). **Deliberately unpinned:** Desmos `v1.9/calculator.js` and `gsi/client` are rolling endpoints redeployed in place by their vendors; a pinned hash would break checkpoints/sign-in on their next release. Version bumps of a pinned library MUST update the matching hash in the same edit.
- **Serverless companion (2026-07-01; hardened + extended 2026-07-03).** The only server-side code in the repository is `src/worker.js`, the Worker entry (`wrangler.jsonc` `main`) serving three API routes in front of the asset pipeline. **`/api/sync`** verifies Google ID tokens with the native Web Crypto API against Google's JWKS and reads/writes per-student progress in the `ODE_PROGRESS_KV` KV namespace; its POST path enforces `application/json`, a double size cap (declared `Content-Length` pre-read + actual text post-read, 128 KiB), and top-level-object shape before the whitelist sanitizer runs; the sanitizer whitelist covers the three core progress keys plus the three telemetry layers (2026-07-10: `ode_events` capped at 1600 packed strings of 64 chars, `ode_skill_state` capped at 600 four-field numeric records, `ode_daily_activity` capped at 400 ISO-date counter pairs). **`/api/contact`** (POST only) is the landing page's lead-generation intake: strict payload shape (`name`/`email`/`message`, trimmed + length-capped, email regex), server-side **Cloudflare Turnstile** validation against `challenges.cloudflare.com/turnstile/v0/siteverify` using the `TURNSTILE_SECRET_KEY` **Worker secret** (provisioned via `npx wrangler secret put TURNSTILE_SECRET_KEY` — never in the repo or `vars`), and verified leads persisted as `lead:<iso>_<rand>` KV records with a 90-day TTL. **`/api/packages`** (2026-07-11, Sprint Rec 5) is the tutoring package entitlement ledger: one endpoint, two **edge-evaluated roles** on the same hybrid bearer. Identity is flat SSO — every verified Google account is a student; the single elevated role is the tutor-owner, granted exclusively by the `OWNER_SUBS` allowlist in `wrangler.jsonc` `vars` (comma-separated Google `sub` claims; subs are opaque public identifiers, not secrets, and gate nothing without a cryptographically verified bearer for the same sub). A student GET prefix-scans only `package:<own sub>:` — other students' records and group rosters are unreachable *by construction*, not by filtering — and every student write is 403. An owner GET returns the full ledger; owner POST actions are `create`, `set-status` (forward-only `quoted → active → completed`), `log-session` (active packages only, 15–480 minute entries, 200-entry cap), `link-student` (relocates an `unassigned` record into a student namespace, copy-then-delete so a fault duplicates rather than loses), and `create-group` (roster of at most 4 validated emails, deduped case-insensitively). Records live at `package:<sub>:<pkgId>` with the reserved non-numeric segments `unassigned` and `group` (Google subs are all-numeric, so collision is impossible). The `PACKAGE_CATALOG` constant is the offering source of truth with grounded Wyzant rate translations — 4-Hour Fundamentals $100 flat, 8-Hour Comprehensive $200 flat, 12-Hour Semester $300 flat (all $25/hr Wyzant equivalent), Small Group Session 8 hours at $120/student ($15/hr equivalent); money never moves through the endpoint, billing stays on Wyzant per-student custom rates. All three routes sit behind a lightweight in-isolate fixed-window **rate limiter** keyed on `CF-Connecting-IP` (sync 60/min, contact 5/min, packages 60/min; 429 + `Retry-After`). Regression surface: `scripts/test_sync_worker.mjs` (`npm run test:sync`), 45 cases. The Worker is a *companion*, not a dependency: the client never requires it to function (Pillar 1 holds — the SPA remains fully `file://` executable).
- **ODE SEO metadata layer (2026-07-04; a routing-config change, logged per the Living Blueprint Rule).** The roadmap SPA is a single static document at `/ode/`, but its curriculum spans 19 first-principles units, so a crawler hitting `/ode/?unit=N` should read `<title>`/`<meta name="description">` written for that unit's concept rather than the generic shell copy. `src/worker.js` owns a localized `ODE_UNIT_SEO` map (keys `"0"`..`"18"`, matching `content/curriculum.json` unit numbers) and a `handleOdeSeo()` handler that fetches the real asset and **stream-rewrites only those two head tags via `HTMLRewriter`** when a recognized `?unit=` selector is present; every miss (no selector, unknown unit, non-GET, non-HTML) returns the untouched asset, so this is a purely additive layer. **The load-bearing routing change:** the handler can only run because `wrangler.jsonc` now lists `"run_worker_first": ["/ode/", "/ode/index.html"]` in `assets`, routing *only* the ODE document through the Worker before the asset server; sub-assets (`/ode/css`, `/ode/js`, `/ode/data`) stay on the direct asset path with zero Worker overhead. SEO copy obeys the §1 constraint (no em-dashes, no ampersands). The SPA's zero-dependency `file://` contract is untouched — the rewrite happens edge-side and `file://` opens never traverse the Worker. **Crawlability (2026-07-05):** the dynamic `/sitemap.xml` handler (`buildSitemapXml()`) now enumerates the two top-level pages **plus one deep-linked `<url>` per unit** (`/ode/?unit=N`, weekly `changefreq`), drawing the unit list from `ODE_UNIT_SEO` itself so a new unit self-registers in the sitemap; every `<loc>` is XML-entity-escaped. So the 19 unit-targeted metadata variants are not just correct-on-hit but independently discoverable by search engines. **Per-unit `<lastmod>` (2026-07-05):** each `ODE_UNIT_SEO` block now carries an ISO-8601 `lastmod` (the git last-commit date of that unit's content package source `scripts/unitN_data.json`), which `sitemapUrlEntry()` emits per unit and falls back to the global `SITEMAP_LASTMOD` baseline only if a block omits it — so the sitemap reports genuine per-unit content history (units 1 and 14-18 predate the Content API v1 batch) rather than one blanket date.
- **Landing-page contact + analytics surface (2026-07-03).** Root `index.html` carries a `#contact` inquiry form wired to `POST /api/contact` by a protocol-guarded inline script (under `file://` the form disables itself and points at the direct booking link — no broken root-absolute fetch). The form embeds the implicit-render Turnstile widget (`.cf-turnstile`, public site key `0x4AAAAAADvT-1PlPvun0Xm_`) and surfaces success/error states through the new `--color-success`/`--color-error` (+ `-soft`) tokens added to the `style.css` `:root` block. **Cloudflare Web Analytics** is injected site-wide on the landing page via the deferred `static.cloudflareinsights.com/beacon.min.js` script (public token `14c519923cd24565bcbd4b336c6bb429`, cookie-free). The `ode/` SPA is untouched by all of this — its zero-dependency `file://` contract is unchanged.
- **Front-page onboarding funnel (2026-07-04).** The hero CTAs are now on-page navigators instead of two identical external links: **"Book Your First Session"** anchors to `#booking` and **"Learn More"** anchors to `#services` (the "Why the Math Confidence Reset?" block). Both scroll smoothly with **zero JavaScript** — they rely on the existing `html { scroll-behavior: smooth }` in `style.css`, so they behave identically over `file://` and `https`. The new `#booking` section ("Booking Your First Session") lays out the three-step Wyzant workflow — **create account → share context → schedule** — as a `.booking-steps` list; the single external booking link (`tinyurl.com/StaplesEducation`, `target="_blank" rel="noopener"`) lives **only** at step 3, the final execution step. All `.booking*` styles are new to `style.css` and reuse the established surface/border/purple-halo tokens (same language as `.card`/`.card-icon`); the copy honors the §1 constraint.

### Pillar 2: Centralized Hash Routing Matrix
- **One entry point.** `ode/js/router.js` exposes `renderCurriculum()`, the sole render function. Every navigation, mode toggle, and checkpoint pass funnels through it, so the view always redraws to match the URL hash.
- **Subpath mounting (2026-07-01 restructure).** In production the SPA mounts at `/ode/`; the hash matrix below is unchanged because every route lives *after* the `#` and every asset link is document-relative, so the router is agnostic to its mount point. Cross-app navigation is one-way by design: the landing page and `reviews.html` link **into** the SPA via the `/ode/` primary CTA in their shared navigation header; the SPA contains no root-absolute links back out, preserving its standalone `file://` contract.
- **History interception** is wired at the bottom of `router.js`:
  - `document.addEventListener("DOMContentLoaded", renderCurriculum)` — initial paint.
  - `window.addEventListener("hashchange", …)` — re-render plus `window.scrollTo(0, 0)` on view changes.
- **The routing matrix** (hash → view):

  | Hash | View | Type |
  |---|---|---|
  | `#practice-sets` | Practice Sets index | static landing |
  | `#cheat-sheets` | Cheat Sheets (PDF) index | static landing |
  | `#quizzes-index` | Quizzes index | static landing |
  | `#interactives` | Interactives hub | static landing |
  | `#dashboard` | Statistics dashboard (Sprint Rec 4: tiles, mastery radar, trend lines, heatmap, session suggestions) | static landing |
  | `#settings` | Profile and Settings split view (student interface always; tutor admin console only when the edge role probe answers `owner`) | static landing |
  | `#practice-sets-N` | Unit N practice set (problems + toggleable solutions) | dynamic sub-route |
  | `#quizzes-index-N` | Unit N quizzes detail | dynamic sub-route |
  | `#unit-N` | Unit N detail (modules, videos, mastery quiz) | dynamic sub-route |
  | `#session-<kind>[~focus]` | Adaptive composed practice (Sprint Rec 3). Kinds: `similar`, `easier`, `harder`, `weak`, `review`, `remedy`; the optional URI-encoded focus after the tilde is a skill id, or `uN` for a remedy detour's unit scope | dynamic sub-route |
  | *(empty / unmatched)* | Table of Contents | fallback |

- **Bookmarkable and back-button-safe** by design: all view state lives in the hash, never in memory. `unitIndexFromHash()` parses `#unit-N`; sub-route handlers bound-check `N` against `CURRICULUM.length` and fall back to the parent index on overflow.
- **Guided-Pathway gating** computes a *global* flat module sequence via `unitFlatOffset()` over `ALL_MODULES`, so checkpoint locking spans unit boundaries (see Pillar 5 / ARCHITECTURE.md §4).

### Pillar 3: The Unified Data Model Separation
- **The canonical content layer (2026-07-03, Content API v1; extended 2026-07-10, Content Schema v2).** All curriculum, quiz, practice, readings, and bank content is canonically owned by the **`content/` tree** — `manifest.json` + `curriculum.json` + per-unit packages `content/units/unit-NN/{guide,practice,quizzes,readings,bank}.json` (`readings.json` required for every unit; `bank.json` optional, the v2 growth surface) — specified by `docs/content-api/CONTENT_API_SPEC.md` (§12 v2 addendum) and its JSON Schemas. The script-injected globals below are **generated artifacts**: `scripts/compile_web.py` compiles `content/` into `ode/js/curriculum-data.js`, `ode/js/readings-data.js`, the 19 lazy chunks `ode/js/bank/bank-unit-NN.js`, and the `ode/data/*.json` mirrors (drift is now impossible by construction; every generated file carries a banner; the compiler also deletes stale chunks and the retired `quiz-data.js` monolith on sight). CI shape: `python scripts/validate_content.py && python scripts/test_content_pipeline.py` (the latter includes `compile_web.py --check` staleness detection — which byte-locks the deterministic dictionary compression — and byte-exact print-reconstruction locks; the compiler's own Node sanity check evaluates every candidate chunk through the real `bank-loader.js` and asserts the hydrated globals match canon counts with all templates expanded). **Edit `content/`, never the generated files.** The print pipeline reads the same canon: `render_latex.py` loads units via `scripts/content_loader.py` (semantic macros `\emph/\strong/\highlight/\work/\warn` lower per target); `build_practice_data.py` is retired. The legacy `scripts/unitN_data.json` files are frozen archival migration sources.
- **Supplemental readings registry (2026-07-10).** `content/units/unit-NN/readings.json` catalogs each unit's Tectonic LaTeX PDFs (`cheat-sheet`, `practice-set`, `reference-guide`, `solutions`, `topic-guide`) with an `https` absolute-URL escape hatch for off-repo documents (future R2-hosted readings). Entries flagged `status: "planned"` (PDF not rendered yet) compile without their `file`, so the UI renders a text label — this retired the 23 dead topic-guide links the old hand-authored catalog produced. Consumers: `AVAILABLE_MATERIALS` in `views-materials.js` is now *derived at parse time* from the generated `READINGS_DATA` global (cheat-sheet entry → primary file, topic guides → sublist), and the unit detail view closes with a Supplemental Readings row (reference-guide primary button + linked list). A non-planned local `file` must exist under `ode/assets/pdfs/` (validator ERROR), so a rendered-PDF/readings mismatch can never ship.
- **Layout never embeds content.** The curriculum map, checkpoint widgets, quiz strings, and practice problems are each isolated in their own script-injected global, consumed by the layout files but never authored inside them:
  - `curriculum-data.js` → `CURRICULUM` (Unit → Module → Video → `interactive_checkpoint`), generated from `content/curriculum.json` (canonical names `videoId`/`checkpoint` are lowered to the legacy `video_id`/`interactive_checkpoint` until the router migrates).
  - `checkpoints/checkpoint-registry.js` + `widgets-unitN.js` → per-unit interactive widget definitions, resolved by the `interactive_checkpoint` key on each module.
  - `bank-loader.js` + `bank/bank-unit-NN.js` → `QUIZ_DATA` (micro-practice keyed by video id; `unit_mastery` keyed by unit number; `pool` — the v2 adaptive bank, per unit) **and** `PRACTICE_DATA` (the web practice store, units 0–18), hydrated lazily per unit (see Pillar 1 lazy bank layer).
  - `readings-data.js` → `READINGS_DATA` (supplemental readings per unit; `AVAILABLE_MATERIALS` derives from it).
- **The renderer is data-driven.** `router.js` reads these globals and builds DOM with shared classes (`.practice-problem-list`, `.practice-solution-list`, `.pdf-download-btn`, `.toc-grid`, …). Adding a unit's content requires no router change — only a data entry.
- **Build-time tooling produces data, never runtime behavior.** `scripts/compile_web.py` (web globals + mirrors, superseding the retired `build_practice_data.py`) and `scripts/render_latex.py` (PDF assets, via `content_loader.py`) are dev-time compilers over the canonical `content/` tree. The web compiler lowers canonical strings into web-clean text (`$…$` / `$$…$$`, semantic macros unwrapped, print spacing stripped) and emits static globals, preserving Pillar 1.
- **The built-sandbox model is rich, not a placeholder.** Where most units fall back to the generic `INTERACTIVE_SANDBOXES` placeholders (one static "coming soon" body per unit), the leading units are served by dedicated programmatic arrays in `router.js`: `UNIT_0_SANDBOXES` (nine entries), `UNIT_1_SANDBOXES` (ten entries), and `UNIT_2_SANDBOXES` (10 entries, the complete first-order toolkit: linear-solver, geometric-transformation, applied-physics, and the master differential-matrix workbench). Each entry carries a stable, **descriptive ID slug** (`unit_0_*`, `unit_1_*`, `unit_2_*` — e.g. `unit_2_integrating_factor_builder`), dashboard copy, and an explicit `render(body)` function rather than an empty body, so the dashboard loop mounts a live, self-contained canvas engine. Every engine runs a **self-terminating `requestAnimationFrame` loop** that stops once its canvas leaves the DOM, uses strict per-card id/variable namespacing (`u0_sN_`, `u1_sN_`, and for Unit 2's cluster `u2_s1_` through `u2_s10_`) so no inner id or state can collide globally, and reads its colors live from the Pillar 4 theme tokens so Light and Dark both render natively. Registration is uniform: each array is folded into `buildInteractiveItems()` under the same `900 + index` sandbox sort band, and each card deep-links by its slug at `#interactives-sandbox-<id>`. The data-driven `isSandbox` path is unchanged — the renderer simply invokes the entry's `render` when present — keeping layout free of engine logic.

### Pillar 4: Dual-Theme Custom Property Framework
- **Tokenized palette, single-attribute switch.** `ode/css/theme.css` declares the light-default palette on `:root` and overrides the *same token names* under `[data-theme="dark"]`. Theme switching is one attribute flip on the root element — never a stylesheet swap (ARCHITECTURE.md §2).

  | Token | Light (`:root`) | Dark (`[data-theme="dark"]`) |
  |---|---|---|
  | `--bg-color` | `#ffffff` | `#121212` |
  | `--text-color` | `#1a1a1a` | `#e0e0e0` |
  | `--panel-bg` | `#f4f4f4` | `#1e1e1e` |

### Pillar 5: Quiz Subsystem Persistence Loop
- **All progress serializes to discrete, namespaced `localStorage` keys** through the `ODEState` IIFE in `ode/js/state.js`. Every key is `ode_*`-prefixed:

  | Key | Holds |
  |---|---|
  | `ode_watched_videos` | array of watched `video_id`s |
  | `ode_passed_checkpoints` | map of `checkpointId → result detail` |
  | `ode_quiz_progress` | map of `quizId → [correctly-answered question ids]` |
  | `ode_learning_mode` | `"exploration"` (default), `"guided"`, or `"adaptive"` (2026-07-11) |
  | `ode_theme_preference` | `"light"` (default) / `"dark"` / `"system"` |
  | `ode_events` | rolling attempt log: packed `"t36\|kind\|0/1\|itemId"` strings, capped at 1500 (telemetry.js, 2026-07-10) |
  | `ode_skill_state` | map of `skillId → { r, a, c, t }` Elo mastery aggregates (rating, attempts, correct, last-seen ms) |
  | `ode_daily_activity` | map of ISO local date → `[events, correct]` counters for the consistency heatmap |

- **Defensive I/O.** `readJSON`/`writeJSON` wrap every access in `try/catch` with a fallback, so corrupt or unavailable storage degrades gracefully instead of throwing.
- **Domain semantics.** A quiz's score *is* the length of its correct-answer array; `clearQuizProgress(quizId)` powers per-quiz "Retry" without touching siblings. `resetAllProgress()` clears watched videos, checkpoints, and quiz progress **but deliberately preserves** `ode_learning_mode` and `ode_theme_preference` (UI preferences survive a progress reset).
- **Telemetry and mastery substrate (2026-07-10, `ode/js/telemetry.js`).** The three telemetry keys above are owned by the `ODETelemetry` IIFE, the event-sourced layer under the statistics dashboard and the adaptive practice engine. Recording surfaces: every graded quiz click in `quiz-engine.js` (wrong answers included — the correct-only `ode_quiz_progress` loop never sees misses), every checkpoint `pass`/`fail`/`guide` in `checkpoint-core.js`, and fresh video completions via `setVideoWatched`. Mastery math is two-sided Elo per skill (seed 1200, expected score `1/(1+10^((d−r)/400))`, K decaying from 48 toward a floor of 8 as attempts accumulate, ratings clamped to [400, 2400]); video events carry no assessment signal and never move a rating. Since Content Schema v2 (2026-07-10) the quiz engine passes each item's authored `skillId` and Elo `difficulty` through to `ODETelemetry.record` when a bank item carries them; curated quiz items fall back to the quiz id and the model's default difficulty of 1200. Retrieval strength decays with a 14-day half-life, computed at read time (`getSkillSnapshot()`), never stored. Bounded by construction: the event log trims to its newest 1500 entries (aggregates have already absorbed older events), skills prune stalest-first past 500, daily counters past 366 days — so the sync payload stays far inside the Worker's 128 KiB cap. Telemetry rides the Pillar 5 loop end to end: `snapshotProgress()` includes all three keys, the boot merge delegates to `ODETelemetry.mergeCloudTelemetry()` (events union by exact packed string re-sorted by decoded time; per skill the record with more attempts wins, ties to the later last-seen stamp; daily counters take the elementwise maximum), and `resetAllProgress()` clears all three. All recording is failure-swallowed: telemetry can never break a lesson.
- **Mode logic (ARCHITECTURE.md §4; extended 2026-07-11).** *Exploration* (default) leaves everything unlocked; checkpoints are optional self-assessments. *Guided Pathway* activates gateways: the next module unlocks only after the current checkpoint passes, enforced via the Pillar 2 flat-offset sequence. On failure the system surfaces a first-principles guiding question, never the answer. *Adaptive Pathway* (Sprint Rec 3) is the Guided gateway sequence plus programmatic **remediation detours**: on every unit detail render, `ODEAdaptive.buildRemediationPanel(unitIndex)` scans the Elo skill state for prerequisite skills from *earlier* units that are weak (rating below 1150) or fading (retention below 0.5 with at least 2 attempts) and, when found, inserts a detour panel above the module list deep-linking a `#session-remedy~uN` composed session scoped to those skills. The mode value rides the standard `ode_learning_mode` loop and the Worker sanitizer whitelists all three values.
- **Adaptive session composer (2026-07-11, Sprint Rec 3, `ode/js/adaptive.js`).** `ODEAdaptive` reads `ODETelemetry.getSkillSnapshot()` plus the schema v2 pools (`QUIZ_DATA.pool`, hydrated on demand through `ODEBank.ensureUnit` — the Pillar 1 lazy contract holds) and composes up-to-8-item QuizEngine configs per kind: `similar` (difficulty within 150 of the skill rating), `easier`/`harder` (at least 50 below/above), `weak` (lowest-rated assessed skills), `review` (retention below the half-life threshold), `remedy` (weak prerequisites of a unit). Unit scoping derives from the skill-id taxonomy (`sk_<n>_slug`, `"N.M"` module titles, `mastery::Unit N`, `micro::<videoId>` via a curriculum-built map); curated mastery items backfill thin pools, stamped with the mastery surface's skill key so session attempts and mastery-quiz attempts move the same Elo rating. The **post-evaluation branching panel** (`buildBranchingPanel`) is appended by the QuizEngine to every Practice Complete summary (runtime-guarded coupling) offering Similar / Easier / Harder / Attack Weak Areas / Spaced Repetition links into `#session-*` routes.
- **Statistics dashboard + chart kernel (2026-07-11, Sprint Rec 4).** `ode/js/charts.js` (`ODECharts`) renders line/radar/bars/heatmap on raw canvas under the sandbox-engine contract: colors read live from the Pillar 4 theme tokens each frame, one self-terminating RAF loop per chart (ends when its canvas leaves the DOM; idle frames redraw only on a token-signature or layout-size change), devicePixelRatio-scaled backing stores. `ode/js/views-dashboard.js` (`#dashboard`) renders stat tiles, a per-unit mastery radar (Elo normalized over the [400, 2400] clamp range), thirty-day trend lines from `ode_daily_activity`, the weakest-skill bars, a 26-week consistency heatmap, and rule-driven suggestion cards that deep-link Rec 3 composed sessions.
- **Profile and Settings split view (2026-07-11, `#settings`, `ode/js/views-settings.js`).** The student interface (profile card, theme and mode toggles, two-step progress reset, `ode_*` cache clear preserving credentials) renders unconditionally and fully offline. The tutor admin console renders **only from the edge's answer**: the view probes `GET /api/packages` through `ODEState.authFetch` (the generalized authenticated fetch extracted from the sync loop — session-token preferred, Google fallback, one 401 retry) and renders whichever role shape returns; the client holds no role logic and a student response physically contains no other student's data. Owner tools: Package Manager (catalog with Wyzant translations, quote creation, activate/complete), Group Session builder, and the session logging ledger. Under `file://` or signed-out the probe never fires and the section explains itself (smoke-suite core F locks this).
- **Cloud sync loop (2026-07-01, optional layer over the localStorage loop; 2026-07-02 long-lived hybrid local/edge token schema).** When a Google Identity Services credential or a Worker-issued edge session is active, `state.js` extends the persistence loop to the `/api/sync` endpoint (root-absolute by necessity — the SPA mounts at `/ode/` while the API lives at the domain root; a protocol guard keeps it inert under `file://`):
  - **Credentials are a two-tier hybrid, both persisted in namespaced `localStorage` keys** (superseding the original session-scoped `sessionStorage` cache, so sign-in survives browser restarts on mobile and desktop):
    | Key | Holds | Lifetime authority |
    |---|---|---|
    | `ode_google_credential` | the raw GIS ID token (RS256 JWT) | Google (`exp` claim, ~1 h; expired tokens dropped on read) |
    | `ode_cloud_session` | `{ token, expiresAt, email }` — the opaque `odesess_*` edge session token | the Worker's KV record (30-day TTL); the local `expiresAt` is only a skip-a-doomed-request hint |
  - **Token exchange:** the first request authenticated with a verified Google ID token is answered with a freshly minted `odesess_*` session token (`session` field on the sync response), which the client adopts and prefers for all subsequent requests. A 401 on a session token clears it and retries once with the Google credential; with neither available the auth UI resets (sign-in surfaces restored via `syncGoogleSurfaces()`) and GIS `auto_select: true` silently re-establishes the account on the next load.
  - **Defensive boot:** `bootCloudSync()` reads the cached session (or a still-valid credential) from `localStorage` on script parse and initializes the cloud loop immediately; corrupt or expired entries fall through to the signed-out state. The GIS callback global is `odeHandleGoogleCredential`; sign-out clears both keys and calls `google.accounts.id.disableAutoSelect()`.
  - **Auth surface lifecycle (2026-07-02).** GIS is initialized **programmatically** (`initGoogleIdentity()` in `state.js`, wired to the SDK's documented `onGoogleLibraryLoad` hook, with a boot-time fallback call). The declarative `g_id_onload`/`g_id_signin` markup was removed: it rendered the embedded button and fired the floating One Tap prompt unconditionally as soon as the async SDK landed, re-showing sign-in chrome to already-authenticated students. `syncGoogleSurfaces()` is the single gate — only a **completely anonymous** visitor (no `ode_cloud_session`, no still-valid `ode_google_credential`) gets the button rendered into `#auth-signin-slot` and a `google.accounts.id.prompt()` One Tap. A signed-in boot hides the slot via the `hidden` attribute, enforced by an `!important` CSS guard (`#auth-signin-slot[hidden]`); a fresh sign-in additionally calls `google.accounts.id.cancel()` to dismiss any floating prompt. Sign-out restores both surfaces (slot unhidden, button rendered on demand, One Tap re-prompted) with `disableAutoSelect()` guaranteeing an account chooser rather than a silent re-login. Every GIS call is wrapped so `file://` and offline sessions no-op silently.
  - **Boot merge:** on activation the cloud record is fetched and **defensively merged** — watched videos and per-quiz correct-answer lists are unioned, checkpoint entries are added only when locally absent (local detail wins) — so a stale device can never erase work done elsewhere. A changed merge triggers one `renderCurriculum()` repaint.
  - **Background push:** every progress mutator (`setVideoWatched`, `setCheckpointPassed`, `setQuizAnswerCorrect`, `clearQuizProgress`, `setLearningMode`, `resetAllProgress`) schedules a silent, debounced (2.5 s) POST of the whitelisted `ode_*` snapshot; `ODEState.requestCloudSync()` exposes the same trigger for widgets that persist through their own pathways. All cloud calls are best-effort `try/catch` — the localStorage loop is authoritative and never blocks on the network.
  - **Server contract:** `src/worker.js` (Pillar 1 serverless companion) intercepts `GET`/`POST` `/api/sync` and resolves the bearer in one of two verification modes before touching any `progress:<sub>` record: an `odesess_*` token validates directly against its KV session record (`session:<sha256(token)>` — only the hash is stored, so a KV dump can never be replayed; TTL expiry yields a 401 challenge), while anything else must pass the full edge Google JWT verification (JWKS signature, `iss`/`aud`/`exp`/`nbf`/`sub`) and is then exchanged for a new 30-day session. Progress writes sanitize every payload against a strict whitelist with size caps. The contract is regression-locked by the committed 45-case cryptographic validation suite `scripts/test_sync_worker.mjs` (real RS256 tokens against a mocked JWKS + TTL-aware KV mock; run `npm run test:sync`).

## 📦 UNIVERSAL SPA CONTENT MATRIX SPECIFICATION (2026-07-07)

The frontend application canvas running under the root assets tree is strictly decoupled from subject-specific content profiles. The interface operates as a stateless structural renderer driven by an active configuration object.

### 1. Invariant Parameters
- **Subject-Neutral Core:** Hard-coded references to specific course sub-paths or fixed dataset dimensions are structurally prohibited within global layout files.
- **Dynamic Scale ($n$):** Content structures must scale fluidly across $n$ structural elements, where $n \in \mathbb{Z}^+$. The DOM mapping engine must dynamically evaluate array length attributes rather than static loop iterations.
- **Taxonomy Abstraction:** Textual display elements for content segments (e.g., "Unit", "Chapter", "Section") must be read directly from the active subject configuration's `structureLabel` property.

### 2. Active Structural Configuration Shapes
Every current and future course manifest must export a unified metadata configuration conforming to the following structure:
- `subjectId`: Lowercase alphanumeric string matching the asset's subdirectory namespace.
- `displayName`: Client-facing plain text string for titles and headings.
- `structureLabel`: The precise linguistic identifier for content groupings.
- `units`: Array containing the structural learning matrices, equations, and interactive canvas components.

### 3. Implementation Mapping (as built 2026-07-07)
The specification above is the durable contract; this subsection records how the ODE shell satisfies it so the two never drift (Living Blueprint Rule).

- **The config object is `SUBJECT_CONFIG`, declared in the hand-authored `ode/js/subject-config.js`** (new file; load order updated in Pillar 1). It is the single master constant the layout reads.
  ```js
  const SUBJECT_CONFIG = {
      subjectId: "ode",            // matches the /ode/ asset namespace and the Worker's KNOWN_SUBJECTS entry
      displayName: "ODE Roadmap",
      structureLabel: "Unit",      // "Chapter"/"Section" for a future subject
      get units() { return typeof CURRICULUM !== "undefined" ? CURRICULUM : []; }
  };
  ```
- **`units` bridges, it does not duplicate.** `CURRICULUM` remains the canonical data global emitted by `scripts/compile_web.py` from the `content/` tree (Pillar 3); `SUBJECT_CONFIG.units` is a live accessor onto it, so the generated-artifact rule is preserved — a new subject swaps its compiled data global and this one config, nothing in the layout.
- **`subjectId` is now the single source of the sync namespace:** `ode/js/state.js` reads `SUBJECT_CONFIG.subjectId` (was a private `SYNC_SUBJECT = "ode"` literal), keeping the client's `progress:<subject>:<sub>` track name and the Worker's `KNOWN_SUBJECTS` registry in agreement from one declaration.
- **`structureLabel` drives every constructed segment label and derived asset path** in the layout files (`router.js`, `views-materials.js`, `views-practice.js`, `views-interactives.js`): the unit-reference / cheat-sheet / practice-set PDF path prefixes and the `"Unit N"` display fallback are now `SUBJECT_CONFIG.structureLabel`-composed. Output is byte-identical for ODE (`structureLabel === "Unit"`), but a `structureLabel` of `"Chapter"` re-labels the shell with zero further edits.
- **`$n$ was already satisfied:** the router and directory views read `SUBJECT_CONFIG.units.length` / `.forEach` (never a literal `19`), so the DOM engine already scales across arbitrary $n$; this refactor added the naming seam, not the iteration dynamism.
- **Verification gate:** `npm run verify` chains `python scripts/validate_content.py`, `python scripts/test_content_pipeline.py`, `npm run test:sync`, `node --check src/worker.js`, and `npx playwright test` (the client smoke suite below). (Windows note: the invocation is `python`, not `python3`, which resolves to the Microsoft Store shim and fails on this toolchain.)

### 4. Client Smoke Suite (2026-07-07, Audit Rec #2)
The universal template player is regression-locked at runtime by a headless Playwright suite — `tests/client-smoke.spec.js`, config `playwright.config.js` (chromium only, no dev server), the last stage of `npm run verify` and also runnable as `npm run test:client`.

- **`file://` fidelity, hermetic execution.** Every test boots `ode/index.html` directly over the `file://` scheme (Pillar 1 contract, no dev server, no `webServer` block) and **blocks all cross-origin (http/https) requests**, so the run is deterministic and internet-free. This is only safe because first-party code guards every CDN global (`typeof renderMathInElement === "function"`, `typeof Desmos === "undefined"`, `gisApi()` null-checks): the shell renders its DOM and text with KaTeX / Desmos / Math.js / GIS absent. Consequence: the console guard's surviving errors are unambiguously first-party.
- **Six assertion cores.** (A) **Console guard** — zero uncaught page exceptions and zero first-party `console.error` on boot and through every navigation, filtering blocked-CDN / Google-GIS / YouTube noise by source. (B) **Route exercise** — walks `#unit-0`, `#unit-1`, and `#unit-{n-1}` (last index read from `SUBJECT_CONFIG.units.length`, never a hardcoded 18) across **desktop / tablet / mobile** viewports, asserting the detail view redraws and the container never overflows the viewport width. (C) **Taxonomy** — drives the Cheat Sheets index, asserts the default copy reads "Open Unit 0 Cheat Sheet", then flips `SUBJECT_CONFIG.structureLabel` to `"Chapter"` live and re-renders, proving the DOM follows the config rather than a literal. (D) **Sandbox mount** — discovers a built engine from the page's own `buildInteractiveItems()` registry, deep-links its `#interactives-sandbox-<id>` route, asserts a `<canvas>` hooks the DOM, then exits the view and asserts every canvas is torn down (the self-terminating RAF contract, observed at the DOM). (E) **Lazy bank** (2026-07-10) — asserts the `QUIZ_DATA` shells are empty at boot, enters `#unit-0` and waits for the mastery card and micro-practice hosts to mount via dynamic chunk injection over `file://`, asserts **only** the visited unit hydrated, checks the Supplemental Readings row, then drives `#practice-sets-2` and asserts the practice list and the schema v2 bank pool hydrate with every parametric template expanded (no `{{` residue). (F) **Capstone routes** (2026-07-11) — asserts `#dashboard` renders its empty state on a fresh profile, then seeds mastery history through the real `ODETelemetry.record` surface and asserts the tiles, chart canvases, and `#session-*` suggestion deep-links mount; drives `#session-weak` and asserts the composed inline quiz hydrates its bank chunks over `file://`; drives `#settings` and asserts the student cards render, the dashboard canvases are gone (the self-terminating RAF contract), the role slot explains itself under `file://`, and the in-view theme toggle flips the Pillar 4 root attribute.
- **Contract:** a change to the shell's boot order, router container-swap, sandbox lifecycle, or the `SUBJECT_CONFIG` seam is **incomplete** until this suite stays green. New subject tracks inherit it unchanged — point `ODE_INDEX` at the new shell (or parametrize it) and the same four cores validate the clone.
